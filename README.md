# solve_omp: High-Performance Parallel Binary Linear Solver

`solve_omp` is a specialized constraint satisfaction solver implemented in C++17 (C++23 recommended). It is designed to find all binary solutions ($x_i \in \{0, 1\}$) for a system of linear equations using a hybrid approach of backtracking search, constraint propagation, and multi-threaded parallelism via OpenMP.

Note: (almost) all of the code and documentation in this repo is AI generated - a mixture of GPT-5.4 and Gemini 3 Flash was used.

## 1. Mathematical Problem Definition

The solver addresses systems of linear equations in the form:

$$\sum_{j=1}^{N} a_{ij} x_j = b_i \quad \text{for } i = 1 \dots M$$

Subject to the constraint:
$$x_j \in \{0, 1\}$$

Where:
*   $N$ is the number of variables (indexed $x_1, x_2, \dots, x_N$).
*   $M$ is the number of equations (rows).
*   $a_{ij}$ are integer coefficients.
*   $b_i$ is the target right-hand side (RHS) for row $i$.

## 2. Core Algorithm

### 2.1. Backtracking Search
The solver performs a Depth-First Search (DFS) through the space of possible assignments ($2^N$). At each step, it selects an unassigned variable and recursively tries both $x_v = 0$ and $x_v = 1$.

### 2.2. Constraint Propagation (Inference)
To avoid exhaustive $2^N$ search, the solver uses **Feasibility Checking** and **Unit Propagation** at every node:

For any row $r$, let $T_r$ be the current target value (RHS minus the contribution of already assigned variables). Let $P_r$ be the sum of all positive coefficients of remaining unassigned variables, and $N_r$ be the sum of all negative coefficients.

1.  **Feasibility Check**: A branch is immediately pruned if $T_r > P_r$ or $T_r < N_r$, as no combination of remaining $0/1$ assignments can satisfy the equation.
2.  **Unit Propagation**: For an unassigned variable $x_v$ in row $r$ with coefficient $c_v$:
    *   If setting $x_v = 0$ would make the row infeasible, $x_v$ is forced to $1$.
    *   If setting $x_v = 1$ would make the row infeasible, $x_v$ is forced to $0$.

### 2.3. Variable Ordering Heuristic
Variables are not picked at random. Before the search begins, they are sorted by a heuristic to prioritize the most "influential" variables first:
1.  **Total Magnitude**: Variables with the largest sum of absolute coefficients across all equations are picked first.
2.  **Degree**: Variables appearing in the most equations are used as tie-breakers.

## 3. Parallel Architecture

### 3.1. Hybrid Task/DFS Strategy
The solver employs a two-tier parallel strategy to maximize CPU utilization while minimizing overhead:

*   **Breadth Phase (`dfs_task`)**: Near the root of the search tree (controlled by `--spawn-depth`), the solver uses `omp task`. This allows OpenMP to dynamically distribute large branches of the search tree across all available cores.
*   **Depth Phase (`dfs_seq`)**: Once the specified depth is reached, the solver switches to a high-performance sequential DFS. This phase uses a **Trail-based Undo mechanism** (stack-based state recovery) instead of copying the entire `WorkerState`, ensuring high cache locality and zero memory allocations during the hot loop.

### 3.2. Lock-Free Statistics
Each thread maintains its own `thread_local` solution count. Global synchronization only occurs at the very end of the program, preventing "cache line bouncing" and atomic contention during the search.

## 4. Input & Output Formats

### 4.1. Input (`equations.txt`)
The input file expects one equation per line. Variables must be prefixed with `x_` followed by a 1-based index. Coefficients can be explicit or implicit.

**Example Format:**
```text
x_1 + x_2 + -x_3 = 1
2*x_1 + -3*x_4 = -1
x_5 + x_6 = 0
```

### 4.2. Binary Output
When `--out-dir` is provided, each thread writes its discovered solutions to a dedicated binary file (`worker_N.bin`). 
*   **Bit-Packing**: Each solution is bit-packed. Variable $x_1$ corresponds to the least significant bit of the first 64-bit word.
*   **Word Size**: Solutions are always aligned to 8-byte boundaries (64-bit words). If you have 100 variables, each solution takes 16 bytes (two 64-bit words).

## 5. Building and Running

### 5.1. Requirements
*   Clang or GCC with OpenMP support.
*   C++23 compatible compiler.
*   `libomp` (for macOS users).

### 5.2. Compilation
You can use the provided `build.sh` script:
```bash
chmod +x build.sh
./build.sh solve_omp
```

Or compile manually:
```bash
g++ -O3 -march=native -fopenmp --std=c++23 solve_omp.cpp -o solve_omp
```

### 5.3. Usage
```bash
./solve_omp [equations_file.txt] [OPTIONS]
```

**Input file:**
Input should be given in text file. One equation per line. Variables denoted with x_1, ..., x_511. Equations do not need to be normalized in any way.

**Options:**
*   `--threads N`: Number of threads (default: hardware concurrency).
*   `--spawn-depth D`: Depth for OpenMP task spawning (default: 8). Increase for very large problems.
*   `--count-only`: Do not write solutions to disk.
*   `--out-dir DIR`: Directory for binary solution files.
*   `--report-every K`: How often to print thread progress (default: 10,000 solutions).

## 6. Performance Tuning Tips

1.  **Spawn Depth**: If your CPU utilization is low, increase `--spawn-depth` (e.g., 10 or 12). If the overhead is too high (slow start), decrease it.
2.  **Variable Ordering**: If the solver is slow, check if your variables are sparse. The heuristic works best when variables appear in multiple equations.
3.  **Binary I/O**: Writing solutions to a fast SSD is recommended if you expect millions of solutions. Using `--count-only` is significantly faster if you only need the total number.
